File D6

The main problem of page 902gdD6.html is that it contains neither place name nor country. So it is impossible to compute timezone offset without restoration.
The code that performs this step is located in src/SerieD6.php

Restoration process is done in two steps because it involves calls to geonames.org web service.
  • The first step copies and reformat the informations of 902gdD6.html to a csv file in directory 2-cura-exported.
    This is done running : 
    php run-gauquelin5.php D6 raw2exported
  • The second step computes place name, country code and timezone offset.
    It can be stopped and started again without re-calling geonames.org web service for records that have already been computed.
    This is done running : 
    php run-gauquelin5.php D6 computeGeo
    For each record, the program loads the csv file, looks at the first record missing geo information, calls the web service, computes the informations and writes again the csv file on disk.
Restoration process worked for all records except for 356 Ruiz Bernardo. This records was completed manually after execution.

Small errors

  • Notice of page 902gdD6.html says the file contains 450 persons, but it contains in fact 449 (NUM 234 is missing).
  • For record 356 Ruiz Bernardo, a check on wikipedia shows that he is born in Orihuela, ES, which shows that there is an error inura file (latitude should be 38N05 instead of 36N05).
  • Once geographical restoration is done, it's possible to check the longitudes and latitudes of cura file.
    This shows that the coordinates are expressed by removing the seconds, and not rounded to the nearest minute, which generates a loss of precision. For example, latitude of 356 Ruiz Bernardo is 00W56 and the real value is 0Â°56'49''.
